# DreamCode 产品需求文档 PRD

版本: v0.1  
日期: 2026-07-08  
阶段: 产品方向定义  
定位: 面向个人用户的本地 CLI/TUI 任务型 Agent, 以学习和复现 Codex、Claude Code、OpenCode 等优秀产品为初期目标。

## 1. 背景和产品定位

DreamCode 的初衷不是先做商业化产品, 也不是一开始追求差异化创新, 而是作为一个学习和实践大规模任务处理型 Agent 的项目。

初期目标是尽可能接近、复现 Codex、Claude Code、OpenCode 这类成熟 agent 产品的核心能力:

- 用户给出一个目标。
- Agent 能理解本地 workspace。
- Agent 能自主规划、搜索、编辑、运行命令、验证结果。
- Agent 能在足够安全的边界内自动执行任务。
- 用户能通过终端 CLI/TUI 观察、介入、审批、恢复和继续任务。

DreamCode 的第一性原则:

> 先复现优秀 coding/task agent 的核心产品能力, 通过真实实现理解 agent runtime、工具调用、上下文管理、权限控制、自动化执行和验证闭环。

因此, 本 PRD 中的设计优先服务于“学习型复现”和“个人本地使用”, 暂不追求企业化、商业化、插件市场、云端协作或产品级差异化。

## 2. 产品目标

### 2.1 核心目标

DreamCode 要成为一个可以在本地目录中工作的 CLI/TUI Agent, 帮助个人用户完成项目工程、代码修改、调研分析、报告撰写等真实任务。

它需要具备以下核心能力:

- 能在用户指定的本地目录中理解项目结构和文件内容。
- 能根据用户目标制定计划, 并把任务拆成可执行步骤。
- 能自主调用工具完成搜索、读取、编辑、命令执行、Web 查询、验证等操作。
- 能在安全兜底机制下支持高自动化 YOLO 模式, 尽量减少用户手动审批。
- 能把每一步行动、输出、文件变更、命令结果记录下来, 便于审计、回放和学习。
- 能以较高性价比完成任务, 在效果、速度、成本之间做合理取舍。

### 2.2 产品原则

1. 复现优先  
   先学习并复现 Codex、Claude Code、OpenCode 已验证的核心能力, 不急于做差异化。

2. 本地优先  
   workspace 就是用户指定的本地目录。第一阶段不把 workspace 泛化成云端空间、SaaS 工作区或企业知识库。

3. 终端优先  
   首要界面是 CLI/TUI。桌面、Web、IDE 插件等形态以后再讨论。

4. 自动化优先  
   默认追求让 agent 自主完成任务。审批机制服务于安全兜底, 不是让用户频繁点确认。

5. 安全兜底  
   YOLO 不是无限权限。DreamCode 必须通过 workspace 限制、危险命令拦截、文件变更记录、外部副作用限制、secret 保护等方式兜底。

6. 可观察  
   Agent 的计划、步骤、工具调用、输出、失败、重试、文件变更都要可见。

7. 可恢复  
   长任务不能因为进程退出或模型失败就彻底丢失。会话、任务和关键状态应可恢复。

8. 性价比敏感  
   DreamCode 面向个人用户, 需要关注 token 成本、模型选择、上下文压缩、工具调用效率和缓存。

## 3. 目标用户和核心场景

### 3.1 目标用户

DreamCode 初期面向个人用户, 尤其是以下几类:

1. 独立开发者  
   希望用 agent 进行完整项目开发。编写代码、排查 bug、跑测试、生成文档
2. 技术型写作者或研究者  
   希望 agent 在本地目录中完成资料搜集、Markdown 报告撰写、项目文档整理。
3. 高自动化倾向用户  
   愿意在安全沙箱和本地目录限制下, 让 agent 尽可能自动执行任务, 只在高风险动作上介入。

### 3.2 非目标用户

DreamCode 初期不面向:

- 企业管理员。
- 大型团队协作用户。
- 非技术普通消费者。
- 需要云端多人协作的组织。
- 需要合规审计、SSO、权限分级、数据治理的企业客户。
- 需要无代码自动化平台的业务用户。

### 3.3 核心场景

#### 3.3.1 项目工程任务

用户在本地项目目录中启动 DreamCode, 让 agent 完成工程类任务。

典型任务:

- 实现一个新功能。
- 修复一个 bug。
- 添加或修复测试。
- 重构某个模块。
- 升级依赖或迁移框架版本。
- 修改配置、脚本、CI 文件。
- 根据报错日志定位问题。
- 运行测试并根据失败继续修复。
- 生成 changelog、release notes、README 更新。

期望体验:

- 用户用自然语言描述任务。
- Agent 阅读相关文件并制定计划。
- Agent 自动编辑文件、运行命令、查看错误、继续修正。
- Agent 最终总结变更、测试结果和剩余风险。

#### 3.3.2 代码库理解和问答

用户希望快速理解一个陌生项目。

典型任务:

- 解释项目结构。
- 找出某个功能入口。
- 分析某个模块如何工作。
- 绘制调用链或数据流。
- 找出潜在风险和技术债。
- 对某个 PR 或 diff 做 review。

期望体验:

- Agent 能用搜索、读取、符号分析等方式探索项目。
- 回答要带文件路径和关键证据。
- 对不确定处明确说明。

#### 3.3.3 调研任务

用户要求 DreamCode 搜集资料并形成结论。

典型任务:

- 调研某个开源库的能力和架构。
- 对比多个技术方案。
- 搜索最新文档并总结使用方式。
- 阅读本地资料和外部网页, 形成结构化结论。

期望体验:

- Agent 能进行 Web 搜索和网页读取。
- Agent 能把来源链接整理到报告中。
- Agent 能区分事实、推断和建议。
- Agent 能把报告保存到 workspace。

#### 3.3.4 报告和文档撰写

用户希望 agent 在本地目录中撰写或更新文档。

典型任务:

- 写 PRD。
- 写技术方案。
- 写调研报告。
- 整理会议纪要。
- 更新 README。
- 把散乱笔记整理成结构化文档。

期望体验:

- Agent 能读已有文档风格。
- Agent 能生成结构清晰的 Markdown。
- Agent 能在写作前先给出大纲。
- Agent 能按用户反馈迭代。

#### 3.3.5 自动化维护任务

用户希望 agent 处理一些重复性本地任务。

典型任务:

- 批量修改文件格式。
- 统一代码风格。
- 清理无用文件或引用。
- 生成索引文档。
- 检查项目健康状态。
- 扫描 TODO/FIXME 并生成任务列表。

期望体验:

- Agent 可以自动执行低风险批量操作。
- 执行前后给出文件变更摘要。
- 对破坏性动作进行拦截或要求确认。

## 4. 功能设计

本章描述 DreamCode 最终希望具备的产品功能形态。具体项目架构、技术栈和阶段 MVP 暂不在本 PRD 中展开。

### 4.1 CLI/TUI 交互界面

DreamCode 的主要入口是终端。

#### 4.1.1 CLI 启动

用户可以在任意本地目录启动 DreamCode:

```bash
dreamcode
dreamcode "修复当前项目的测试失败"
dreamcode --mode yolo "实现 README 中描述的功能"
dreamcode resume
```

基础能力:

- 默认以当前目录作为 workspace。
- 支持通过参数指定 workspace。
- 支持直接传入任务 prompt。
- 支持恢复上一次任务。
- 支持查看历史任务。
- 支持设置运行模式。

#### 4.1.2 TUI 主界面

TUI 需要展示:

- 当前任务目标。
- Agent 当前状态。
- 计划和 todo 列表。
- 正在执行的步骤。
- 工具调用记录。
- 命令输出摘要。
- 文件变更摘要。
- 等待用户审批的问题。
- 最终结果和验证状态。

交互能力:

- 输入新的用户指令。
- 中断当前运行。
- 继续运行。
- 修改计划。
- 批准或拒绝高风险动作。
- 查看某一步详细日志。
- 查看文件 diff。
- 恢复或回滚文件变更。

### 4.2 工作区管理

DreamCode 的 workspace 只表示用户的本地目录。

#### 4.2.1 工作区定义

workspace 是 agent 可以理解和操作的根目录。

规则:

- 默认 workspace 为启动命令时的当前目录。
- Agent 默认只能读写 workspace 内文件。
- 访问 workspace 外文件需要明确授权或默认禁止。
- `.gitignore`、`.dreamcodeignore` 等规则可影响上下文收集。
- `.env`、secret 文件、密钥文件默认不可读或需要特殊审批。

#### 4.2.2 项目理解

Agent 需要能识别:

- 文件树。
- 项目语言和框架。
- 包管理器。
- 构建命令。
- 测试命令。
- lint/format 命令。
- Git 状态。
- README、文档、配置文件。
- 项目规则文件。

#### 4.2.3 工作区索引

DreamCode 应维护轻量索引:

- 文件路径索引。
- 文本搜索索引。
- 最近读取文件。
- 重要文件摘要。
- 项目命令摘要。
- 可选的代码符号索引。

索引用于提升搜索、上下文选择和任务执行效率。

### 4.3 Agent 工作循环

DreamCode 的 agent loop 需要接近 Codex、Claude Code、OpenCode 的工作方式。

核心循环:

1. 理解目标。
2. 收集上下文。
3. 制定计划。
4. 调用工具执行步骤。
5. 观察结果。
6. 根据结果调整计划。
7. 验证任务是否完成。
8. 输出总结。

要求:

- Agent 不应只给建议, 而应尽可能实际执行。
- Agent 需要主动读取文件、搜索项目、运行命令。
- Agent 需要在失败时继续诊断和修复。
- Agent 需要知道何时向用户提问。
- Agent 需要避免无意义循环。

### 4.4 运行模式

DreamCode 需要提供多种运行模式, 但产品倾向是优先支持安全兜底下的自动化 YOLO。

#### 4.4.1 规划模式

只读规划模式。

允许:

- 读取文件。
- 搜索文件。
- Web 查询。
- 分析项目。
- 输出计划。

禁止:

- 修改文件。
- 执行会改变环境的命令。
- 安装依赖。
- 删除文件。

适用场景:

- 大任务执行前先让 agent 理解项目。
- 用户想先看方案。
- 高风险任务预研。

#### 4.4.2 引导模式

引导模式。

行为:

- 读操作自动执行。
- 文件编辑需要用户批准。
- 命令执行按风险分级。
- 危险动作需要确认。

适用场景:

- 用户还不完全信任 agent。
- 重要项目。
- 需要较多人工控制。

#### 4.4.3 安全 YOLO 模式

安全兜底自动化模式, 是 DreamCode 的重点模式。

行为:

- workspace 内低风险读写自动执行。
- 常见测试、构建、lint、format、git diff 等命令自动执行。
- Agent 可以自动编辑文件、运行验证、继续修复。
- 危险命令、外部副作用、secret 访问、workspace 外写入仍然拦截。

自动允许的典型动作:

- 读取 workspace 内普通文件。
- 新增或修改 workspace 内代码和文档。
- 运行 `npm test`、`pytest`、`cargo test`、`go test` 等测试命令。
- 运行 lint、format、typecheck。
- 运行只读 git 命令。
- 使用搜索工具。
- 访问公开网页。

需要拦截或确认的典型动作:

- 删除大量文件。
- 修改 workspace 外文件。
- 读取 `.env`、SSH key、token、credential 文件。
- 执行 `rm -rf`、磁盘清理、权限修改等危险命令。
- 安装全局依赖。
- 执行网络发布、部署、支付、发送消息等外部副作用。
- 推送 git commit 或远程分支。

Safe YOLO 的定义:

> 尽可能自动, 但不跨越本地 workspace、安全文件、危险命令和外部副作用边界。

#### 4.4.4 完全访问模式

高信任模式。

行为:

- 尽可能少审批。
- 允许更广泛的命令和文件操作。
- 仍保留最低限度的危险动作提醒。

适用场景:

- 隔离环境。
- 临时测试目录。
- 用户明确愿意承担风险。

### 4.5 工具系统

DreamCode 需要内置一组基础工具, 供 agent 在任务中调用。

#### 4.5.1 文件工具

能力:

- 读取文件。
- 按行读取文件。
- 创建文件。
- 修改文件。
- 应用 patch。
- 删除文件。
- 移动或重命名文件。
- 查看 diff。

要求:

- 文件写入必须记录变更。
- 编辑失败要能回退或重试。
- 大文件读取需要截断和分页。
- 二进制文件默认不直接读入上下文。

#### 4.5.2 搜索工具

能力:

- 按文件名 glob 搜索。
- 按内容 regex 搜索。
- 快速全文搜索。
- 排除 ignored 文件。
- 搜索历史结果复用。

#### 4.5.3 Shell 工具

能力:

- 执行 shell 命令。
- 设置工作目录。
- 设置超时时间。
- 获取 stdout/stderr/exit code。
- 支持长命令中断。
- 支持常见交互命令的安全处理。

要求:

- 命令执行必须经过权限引擎。
- 输出需要摘要和完整日志引用。
- 超时后要能停止进程。
- 危险命令要拦截。

#### 4.5.4 Git 工具

能力:

- 查看状态。
- 查看 diff。
- 查看 log。
- 查看分支。
- 创建分支。
- stage 文件。
- 生成 commit message。

默认限制:

- 不自动 push。
- 不自动 force push。
- 不自动改写历史。
- commit 是否自动执行由运行模式决定。

#### 4.5.5 Web 工具

能力:

- Web 搜索。
- 获取网页内容。
- 总结网页。
- 提取来源链接。
- 下载公开文档。

要求:

- 调研类任务必须保留来源。
- 对动态或时效性事实要标注查询日期。
- 避免把网页大段原文直接塞入上下文。

#### 4.5.6 用户提问工具

Agent 需要能在必要时向用户提问。

适用场景:

- 需求不明确。
- 多个实现方向差异较大。
- 需要用户提供凭证或业务判断。
- 风险动作需要确认。

要求:

- 提问应尽量少而关键。
- 能给出默认推荐选项。
- 用户不回答时, 在低风险场景可按合理假设继续。

#### 4.5.7 Todo/计划工具

Agent 需要维护任务计划和进度。

能力:

- 创建 todo。
- 更新 todo 状态。
- 标记阻塞。
- 记录验证步骤。
- 生成最终总结。

TUI 应直接展示 todo 状态。

#### 4.5.8 可扩展工具接口

最终形态中, DreamCode 应允许用户增加自定义工具。

但初期设计目标只是支持学习和复现, 不要求做复杂插件市场。

可扩展方式:

- 本地脚本工具。
- MCP server 工具。
- 配置文件声明工具。

### 4.6 权限和安全兜底

权限系统是 DreamCode 支持 YOLO 自动化的前提。

#### 4.6.1 权限决策

每个工具调用都应经过权限判断。

判断维度:

- 工具类型。
- 操作类型。
- 目标路径。
- 是否在 workspace 内。
- 是否涉及 secret。
- 是否可能删除或覆盖大量内容。
- 是否有外部副作用。
- 当前运行模式。
- 用户配置规则。

决策结果:

- allow: 自动执行。
- ask: 等待用户批准。
- deny: 直接拒绝。

#### 4.6.2 危险命令拦截

默认拦截或要求确认:

- 递归删除。
- 格式化磁盘。
- 修改系统权限。
- 删除 git 历史。
- 强推远程分支。
- 清空数据库。
- 部署生产环境。
- 发送外部消息。
- 访问敏感文件。

#### 4.6.3 Secret 保护

规则:

- 默认不读取 `.env`。
- 默认不读取 SSH key。
- 默认不读取 credential/token 文件。
- 工具输出中出现疑似 secret 时需要脱敏。
- Agent 不应把 secret 写入日志、报告或 prompt。

#### 4.6.4 文件变更保护

能力:

- 记录每次文件写入前后的 diff。
- 支持查看变更。
- 支持撤销单次变更。
- 支持任务级回滚。
- 对批量修改提供摘要。

#### 4.6.5 循环和失控保护

能力:

- 最大工具调用次数。
- 最大连续失败次数。
- 最大命令运行时间。
- 最大 token/成本预算。
- 检测重复执行相同命令或相同编辑。
- 失控时暂停并向用户报告。

### 4.7 上下文管理

DreamCode 需要高效管理上下文, 保证效果、速度和成本。

#### 4.7.1 项目规则

支持在 workspace 中读取 DreamCode 专属规则文件, 例如:

- `DREAMCODE.md`
- `.dreamcode/rules.md`
- `.dreamcode/config.toml`

规则内容可包括:

- 项目结构说明。
- 常用命令。
- 代码风格。
- 测试要求。
- 禁止操作。
- 用户偏好。

#### 4.7.2 上下文选择

Agent 应主动选择相关上下文, 而不是把整个项目塞给模型。

能力:

- 按任务搜索相关文件。
- 读取关键片段。
- 摘要长文件。
- 复用已读内容。
- 对上下文进行压缩。
- 在回答中引用文件路径。

#### 4.7.3 会话记忆

DreamCode 应保存当前会话中的重要信息:

- 用户目标。
- 已读文件。
- 已执行命令。
- 已修改文件。
- 已失败尝试。
- 已确认规则。

长期记忆可以作为后续能力, 但初期不应过度设计。

### 4.8 会话、历史和恢复

DreamCode 需要保存任务历史。

能力:

- 创建新任务。
- 保存任务 transcript。
- 恢复上次任务。
- 查看历史任务。
- 继续某个任务。
- 中断当前任务。
- 分叉任务作为可选能力。

每个任务记录:

- 用户输入。
- Agent 回复。
- 计划。
- 工具调用。
- 命令结果。
- 文件变更。
- 审批记录。
- 最终总结。

### 4.9 验证和完成判断

DreamCode 的任务完成不能只依赖模型自我判断。

#### 4.9.1 验证方式

编码任务:

- 运行测试。
- 运行 typecheck。
- 运行 lint。
- 运行 build。
- 检查 git diff。
- 检查目标文件是否修改正确。

调研任务:

- 检查来源链接。
- 检查结论是否覆盖用户问题。
- 检查是否区分事实和推断。

文档任务:

- 检查文件是否生成。
- 检查结构是否完整。
- 检查是否符合用户指定格式。

#### 4.9.2 最终总结

任务结束时必须输出:

- 完成了什么。
- 修改了哪些文件。
- 运行了哪些验证。
- 验证结果如何。
- 有哪些风险或未完成事项。
- 下一步建议。

### 4.10 性能和成本控制

DreamCode 面向个人用户, 必须关注性能和性价比。

能力:

- 支持配置模型。
- 支持为不同步骤选择不同模型。
- 支持上下文压缩。
- 支持工具结果摘要。
- 支持缓存网页和文件摘要。
- 支持限制最大 token、最大费用或最大步骤数。
- 支持显示本次任务消耗。

产品要求:

- 简单任务不能过度消耗强模型。
- 长任务要有进度和预算提示。
- 失败重试要有限制。

### 4.11 输出产物

DreamCode 的产物包括:

- 代码修改。
- 测试修改。
- 配置修改。
- Markdown 文档。
- 调研报告。
- PRD 草稿。
- 任务总结。
- Git diff。
- commit message 草稿。

产物要求:

- 保存在 workspace 内。
- 可被用户查看和编辑。
- 关键产物应在最终总结中列出路径。

### 4.12 配置系统

DreamCode 需要支持用户配置。

配置范围:

- 默认模型。
- 默认运行模式。
- 权限规则。
- Web 搜索开关。
- 最大预算。
- 最大步骤数。
- 是否允许 workspace 外读取。
- 忽略文件规则。
- 常用命令。

配置层级:

- 全局用户配置。
- workspace 配置。
- 单次命令参数。

优先级:

单次命令参数 > workspace 配置 > 全局用户配置 > 默认值。

### 4.13 与现有优秀产品的能力对齐

DreamCode 的最终形态应尽量对齐以下能力:

Codex 类能力:

- 本地 workspace 执行。
- 线程/任务恢复。
- 自动读写文件。
- 命令执行。
- 沙箱和审批。
- SDK/API 后续可考虑。

Claude Code 类能力:

- Plan mode。
- 自动执行和人工介入。
- 项目规则文件。
- hooks/skills 后续可考虑。
- 会话历史。
- 任务中断和继续。

OpenCode 类能力:

- TUI 优先。
- 多模型配置。
- 权限规则。
- 内置工具清晰化。
- agents/modes 配置。
- MCP server 接入。

## 5. 非目标

DreamCode 当前阶段明确不做或不优先做以下内容。

### 5.1 商业化

不考虑:

- 订阅套餐。
- 计费系统。
- 支付。
- 企业销售。
- 插件市场变现。

### 5.2 企业功能

不考虑:

- 多人协作。
- 企业 SSO。
- RBAC。
- 管理后台。
- 合规审计平台。
- 数据保留策略。
- 企业知识库治理。
- 组织级策略下发。

### 5.3 云端优先

不优先:

- 云端长任务。
- 托管 workspace。
- 云端容器。
- 远程 VM。
- 多设备同步。

DreamCode 初期以本地运行和本地目录为核心。

### 5.4 大而全的通用自动化平台

不追求一开始成为:

- Zapier 类自动化平台。
- RPA 平台。
- 企业工作流引擎。
- 桌面操作机器人。
- 全能办公助手。

### 5.5 产品差异化

初期不以差异化为目标。

暂不优先:

- 独特 UI 创新。
- 行业垂直方案。
- 商业插件生态。
- 企业连接器矩阵。
- 与 Codex/Claude Code/OpenCode 明显不同的新概念。

### 5.6 复杂协作和发布能力

暂不优先:

- 自动部署生产环境。
- 自动 push 远程分支。
- 自动创建线上 PR。
- 自动发送邮件或消息。
- 自动修改生产数据库。

这些动作如果未来支持, 也必须默认要求人工确认。

## 6. 成功指标

DreamCode 的成功不以商业收入衡量, 而以学习效果、功能完整度、任务完成质量、安全性、自动化程度和性价比衡量。

### 6.1 学习目标指标

- 能清楚解释并实现 agent loop。
- 能清楚解释并实现 tool call runtime。
- 能清楚解释并实现 permission engine。
- 能清楚解释并实现 context management。
- 能清楚解释并实现 session persistence。
- 能通过实现过程理解 Codex、Claude Code、OpenCode 的核心设计。

### 6.2 功能完整度指标

DreamCode 应能完成以下端到端任务:

- 在本地项目中修复一个测试失败。
- 在本地项目中实现一个小功能。
- 在本地项目中完成一次代码重构并运行验证。
- 阅读一个陌生项目并输出结构说明。
- 搜索外部资料并生成 Markdown 调研报告。
- 根据用户要求撰写一份 PRD 或技术方案。
- 在 Safe YOLO Mode 下自动完成低风险任务。
- 在危险动作出现时正确拦截或请求确认。

### 6.3 任务质量指标

- 编码任务最终有可查看 diff。
- 编码任务尽量运行测试或说明无法测试原因。
- 调研任务包含来源链接。
- 文档任务结构完整、可直接保存为 Markdown。
- 最终总结能准确说明完成内容、验证结果和风险。

### 6.4 自动化指标

- 在 Safe YOLO Mode 下, 低风险本地编码任务大部分步骤无需人工审批。
- Agent 能自主处理常见失败, 如测试失败、类型错误、命令报错。
- Agent 能在有限轮次内停止无效尝试并汇报阻塞原因。

### 6.5 安全指标

- 默认不读取 secret 文件。
- 默认不写 workspace 外文件。
- 默认不执行高危删除命令。
- 默认不产生外部副作用。
- 所有文件修改都有记录。
- 用户能查看和撤销文件变更。

### 6.6 性能和性价比指标

- 小任务启动和响应速度可接受。
- 文件搜索和读取不应明显卡顿。
- 长任务有进度反馈。
- 支持限制最大步骤数和成本。
- 能通过上下文选择和摘要降低 token 消耗。

### 6.7 体验指标

- 用户能用一句自然语言启动任务。
- 用户能看懂 agent 正在做什么。
- 用户能随时中断任务。
- 用户能恢复历史任务。
- 用户能知道为什么某个动作被拦截。
- 用户能从最终总结中快速判断任务是否完成。

## 7. 后续待定问题

以下问题留到后续“项目架构、技术栈、阶段 MVP”设计中确定:

- 使用 TypeScript-first 还是 Rust core + TypeScript UI。
- CLI/TUI 具体框架选择。
- 模型 provider 抽象方式。
- 是否首版支持 MCP。
- 是否首版支持 LSP。
- session store 使用文件、SQLite 还是其他方案。
- permission engine 的具体规则格式。
- patch/edit 工具的实现方式。
- Web 搜索和网页读取能力如何接入。
- Safe YOLO Mode 的默认规则集。
- 每个阶段的 MVP 切分。
